# 项目最精简架构与启动说明

## 1. 项目目标

本项目用于演示并观察：不同无线链路参数（模式、Eb/No）下，远程操控机械臂的响应变化。

## 2. 当前完成情况

- 已打通完整闭环：前端控制 -> 后端无线链路仿真 -> 机械臂动作 -> 指标回传。
- 已支持无线实验预设与手动参数：`Basic AWGN`、`Advanced CDL+OFDM`、`Eb/No`。
- 已支持关键观测指标：平均 BER、单次无线处理时延、轨迹误差均值、累计无线时延、命令队列长度。
- 已支持网络拓扑地图展示：路由器位置、链路时延、路由器繁忙度、排队深度。
- 已支持 Docker 单命令启动。
- 当前指标为仿真链路统计，不是公网真实链路测量值。

## 3. 最精简架构（按职责）


## 4. 各部分功能说明

### 4.1 后端 `backend/app/main.py`

- 提供 `GET /health` 健康检查。
- 提供 `WebSocket /ws` 实时双向通信。
- `receiver()`：接收控制命令与无线实验参数。
- 控制命令支持：`robot_joint_control`（关节目标）与 `robot_ee_control`（末端目标，后端轻量IK逆解）。
- 无线仿真中间层：
	- 命令队列与传输调度
	- 关节角比特化传输（float32 编码/解码）
	- BER 统计（按比特翻转结果计算）
	- 模式切换（Basic AWGN / Advanced CDL+OFDM）
	- Eb/No 参数化
- `command_processor()`：仅在命令经过仿真后才应用到机械臂关节。
- `ack` 回传：回传 `cmd_seq`、BER、单次无线处理时延、轨迹误差。
- `sender()`：持续推送 telemetry（无线状态、实验配置、队列长度、机械臂状态）。
- 新增 `GET /api/v1/experiments/export/csv`：导出近期实验记录（可直接用于论文绘图）。

### 4.2 前端 `frontend/src/App.tsx`

- 控制区：末端目标控制（IK）、无线实验参数、复位。
- 无线实验区：一键切换预设，或手动调整 `mode/EbNo/力传感标志`。
- 通信区：
	- 发送控制命令（含 `cmd_seq`、`client_timestamp`）
	- 接收 `ack` 与 telemetry
	- 展示 BER、无线处理时延、轨迹误差、运行时长
- 可视化区：React Three Fiber 渲染 3D 机械臂。
- 仪表盘区：显示 BER、无线链路处理时延、轨迹误差、实验运行状态与队列积压。
- 网络地图区：显示路由器位置、链路质量、节点繁忙度，并给出可扩展优化方向。

### 4.3 部署层

- `docker-compose.yml`：统一编排前后端容器。
- `frontend/Dockerfile`：前端构建后由 Nginx 提供静态服务。
- `backend/Dockerfile`：运行 FastAPI + Uvicorn。

## 5. 启动方式（唯一推荐）

先确认 Docker Desktop 已启动，在项目根目录执行：

```bash
docker compose up --build
```

访问：

- 前端：`http://localhost:8080`
- 后端健康检查：`http://localhost:8000/health`

停止：

```bash
docker compose down
```

## 6. 核心运行链路

- 前端按钮发送命令 -> 后端无线仿真层处理 -> 命令出队后机械臂动作 -> 后端回传 `ack` 和 telemetry -> 前端刷新 3D 与仪表盘。

## 7. 如何观察实验现象

- `高 Eb/No`：BER 更低、轨迹误差更小、控制更平滑。
- `低 Eb/No`：BER 上升、轨迹误差增大，可能出现动作偏差。
- `Advanced CDL+OFDM`：通常 BER 更低，但单次无线处理时延更高。

## 8. 分步改造路线（模型资产 + 运动逻辑）

### 第1步（已完成）

- 前端增加模型资产检测与自动加载：
	- 存在 `/assets/robot/ur5e_obj/base_0.obj` -> 使用真实模型渲染。
	- 不存在 -> 回退到内置几何机械臂。
- 目标：不影响现有闭环、不增加启动复杂度，保持 Docker 单命令可用。

### 第2步（已完成）

- 后端已将 `robot_joint_control` 的目标值改为轨迹分段（waypoint）执行。
- `receiver()` 继续接收原有消息格式，支持 payload 中 `steps` 参数（默认 8，范围 1~60）。
- `command_processor()` 在命令出队后生成插值路径，`motion_stepper()` 按固定周期逐点执行。
- 预期收益：减少“瞬时跳变”，提升网络劣化场景下的动作连续性。

### 第3步（已完成，轻量IK版本）

- 前端控制区新增末端目标输入（X/Y/Z）与腕部俯仰角，发送 `robot_ee_control` 命令。
- 后端对末端目标做轻量逆解（简化平面几何 IK + 基座偏航），再映射到关节空间。
- 逆解结果复用第2步的 waypoint 执行链路，保持网络仿真与轨迹队列机制一致。

### 第4步（已完成，可选MuJoCo模式）

- 后端引入 PhysicsEngine 抽象层：LightweightPhysicsEngine（默认线性插值）、MuJoCoPhysicsEngine（可选）。
- 通过环境变量 `PHYSICS_ENGINE` 在两种模式间切换（默认 lightweight；设置 mujoco 启用 MuJoCo）。
- 架构特性：
  - 若 MuJoCo 不可用或安装失败，自动回退到 lightweight 模式。
  - 协议保持兼容：前端无需改动，所有命令消息格式不变。
  - WebSocket 接口、网络仿真、汽车队列机制完全兼容两种模式。
- 启用方法：
  1. 一次性启动（推荐）：
	  - PowerShell：`$env:INSTALL_MUJOCO="true"; $env:PHYSICS_ENGINE="mujoco"; docker compose up --build`
	  - Bash：`INSTALL_MUJOCO=true PHYSICS_ENGINE=mujoco docker compose up --build`
  2. 分步启动（等价）：先 `docker compose build --build-arg INSTALL_MUJOCO=true backend`，再设置 `PHYSICS_ENGINE=mujoco` 后 `docker compose up`。
  3. 说明：`docker compose` 不支持 `-e` 短参数（`docker compose -e ...` 会报错）。
  4. 若使用 `up --build` 启动 MuJoCo，必须同时设置 `INSTALL_MUJOCO=true`，否则会按默认值重建为 lightweight 依赖。
- 预期收益：可选启用物理仿真引擎，保持轻量部署路径不变；为后续完整运动学模型或力学仿真奠定基础。
